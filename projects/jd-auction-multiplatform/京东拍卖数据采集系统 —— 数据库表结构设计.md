## 一、系统概览

| 项目       | 说明                                                         |
| :--------- | :----------------------------------------------------------- |
| 数据源     | 京东资产拍卖平台 (paimai.jd.com) 及其他拍卖平台              |
| 资产类型   | 10 大类：土地、房地产、设备、车辆、债权、股权、知识产权、物资产品、用益物权、其他 |
| 采集方式   | JSON API 结构提取 + HTML 表格解析 + AI（百炼）语义提取       |
| 正式数据库 | **MySQL 8.0+**，InnoDB，字符集 utf8mb4，数据库名 `auction_data` |
| 调试备用   | SQLite，仅用于单次实验、临时调试、CSV 导出                   |
| 数据库驱动 | pymysql，DictCursor，utf8mb4 字符集                          |

## 二、架构与数据流

```
京东 API / HTML / 附件 / AI
        ↓
jd_scraper_v2 提取与标准化
        ↓
MySQLJDScraperDatabase 直接写入
        ↓
auction_data MySQL 正式表
        ↓
Viewer / 调度平台 / Navicat 查询
```

**设计原则**：

1. **MySQL 是唯一正式数据库**，采集器直接写入，不再经过 SQLite 中转。
2. **SQLite 仅作为调试备用**，通过命令行 `--backend sqlite` 切换。
3. **三段式字段模型**：金额、面积、日期类字段分为业务值字段（`DECIMAL/DATE`，用于排序统计）+ 展示值字段（`VARCHAR`，用于页面展示）+ 证据字段（`field_extractions` 表，用于追溯来源）。
4. **字段证据统一进 `field_extractions`**，主表不冗余存储溯源信息。
5. **原始接口/HTML 统一进 `raw_payloads`**，主表不保存原始快照。
6. **不依赖 JSON 快照字段**，不再使用 `common_fields_json / special_fields_json` 作为查询依赖。

## 四、核心设计决策

| 决策 | 原因 |
| :--- | :--- |
| MySQL 为唯一正式数据库 | SQLite 不适合并发写入和多平台支持；MySQL 支持完整的 DDL 类型约束和事务隔离 |
| 三段式字段模型（业务值+展示值+证据） | 金额/日期/面积的"展示格式"和"排序数值"互斥，分开存储各司其职 |
| 共有字段统一一张表 | 10 类资产共有的字段（状态、价格、时间等）不重复定义 |
| 特有字段每种类型独立表 | 每类资产关注点不同，独立表可灵活扩展 |
| JSON API + AI 双重提取 | JSON 结构化数据优先；内容理解交给 AI 语义提取 |
| 原始数据全量保存（raw_payloads） | 支持事后回溯、数据修正、手动补采 |
| 字段溯源表（field_extractions） | 每个字段记录来源、方法、置信度、标准化值，全链路可审计 |
| field_extractions 增加分类标准化列 | 增加 `numeric_value/date_value/datetime_value`，支持直接 SQL 聚合查询 |
| 一对多用明细表（debt_details / ip_details） | 解决多户债权包、多项知产的"压扁"问题 |
| 不依赖 JSON 快照字段 | 主表字段已完整展开；field_extractions 已保存完整提取记录 |
| 处置方与处置机构分离（disposal_party / disposal_agency） | 法院/AMC（真正处置方）≠ 店铺/拍辅机构（服务机构） |
| 金额统一以"元"为单位 | 避免万元/亿元混用导致分析出错 |
| 附件与媒体分离存储 | 文件附件（PDF/Excel）≠ 标的图片/视频 |
| AI 图片 OCR 辅助提取 | IP 知产表格、债权表格等多在图片附件中，需 AI vision 逐行识别 |
| 采集失败不阻断 | partial_success 状态允许部分成功，单个标的失败不影响整体批次 |
| project_status 入库保存快照 | 每次采集/增量刷新时重新计算并覆盖；页面可辅助实时判断 |

## 五、表结构详细设计

### 5.1 crawl_batches — 采集批次表

| 字段 | 类型 | 说明 | 备注 |
| :--- | :--- | :--- | :--- |
| batch_id | VARCHAR(64) PK | 批次唯一标识 | UUID 格式 |
| started_at | DATETIME | 批次开始时间 | 采集启动时间戳 |
| finished_at | DATETIME | 批次完成时间 | 最后一个标的处理完毕的时间 |
| parameters_json | LONGTEXT | 采集参数 JSON | 类目、数量、筛选条件等 |
| status | VARCHAR(30) | 状态：running/success/partial_success/failed | partial_success = 部分标的采集失败 |
| message | TEXT | 批次消息 | 错误信息或批次说明 |
| summary_json | LONGTEXT | 批次统计、错误和参数汇总 JSON | 成功数、失败数、扫描数等 |

**`summary_json` 字段内容示例**：

```json
{
    "categories": ["land", "real_estate", "debt", "equipment", "vehicle", "equity", "ip", "goods", "usufruct", "other"],
    "per_category_limit": 5,
    "total_scanned": 95,
    "total_collected": 42,
    "success_count": 38,
    "failed_count": 4,
    "failures": [
        {"paimai_id": "310xxx", "error": "API timeout"}
    ],
    "message": "部分标的因网络超时未能完成采集"
}
```

> **设计演进**：v1 版本将 `parameters_json` 和 `message` 分开存储；v2 增加 `summary_json` 将参数、统计、异常统一管理，保留前两者用于兼容。

### 5.2 raw_payloads — 原始数据存档表（1条标的 = 1行）

| 字段 | 类型 | 说明 | 备注 |
| :--- | :--- | :--- | :--- |
| paimai_id | VARCHAR(32) PK | 京东拍卖标的 ID | 与 auction_items_common 一一对应 |
| batch_id | VARCHAR(64) | 所属批次 | FK → crawl_batches |
| source_url | VARCHAR(500) | 标的页面 URL | 页面访问链接，可直接拼出 |
| list_json | LONGTEXT | 列表页 API 原始响应（JSON） | 包含摘要信息，数据量小 |
| detail_json | LONGTEXT | 详情页 `getWareCoreDataBff` API 响应 | 结构化字段的主数据源 |
| product_basic_json | LONGTEXT | 商品基础信息 API 原始 JSON | 常包含竞价起止时间、评估价、处置/上传机构等 |
| realtime_json | LONGTEXT | 实时出价数据 `getPaimaiRealTimeData` API 响应 | 竞价中的实时价格，仅未成交标的有值 |
| description_html | LONGTEXT | 标的详情 HTML 原文 | AI 文本提取的来源 |
| notice_html | LONGTEXT | 竞买须知 HTML 原文 | 竞拍规则与流程说明 |
| announcement_html | LONGTEXT | 竞买公告 HTML 原文 | 含资产评估、瑕疵等关键信息 |
| attachments_json | LONGTEXT | 附件文件列表（JSON） | 文件名与下载 URL |
| attachment_texts | LONGTEXT | 附件解析后的文本内容（JSON） | 已从 PDF/Word 提取的纯文本 |
| vendor_json | LONGTEXT | 处置机构信息 API 响应 | 法院/AMC/拍卖行等信息 |

## 七、字段类型规范化

### 7.1 设计原则

MySQL 8.0+ 对字段类型执行严格约束。定义规范类型有三重价值：

1. **查询性能**：`DECIMAL` 字段可以直接做范围查询和聚合函数，VARCHAR 需要 CAST
2. **ORM 映射**：SQLAlchemy 等 ORM 框架能正确推断列类型
3. **文档自描述**：看建表语句就知道每列存什么，降低理解成本

### 7.2 类型规范总则

| 字段内容 | 推荐类型 | 典型长度 | 示例 |
| :--- | :--- | :--- | :--- |
| ID / 主键 | VARCHAR(32) | 32 | `paimai_id` |
| 短文本标签 | VARCHAR(100) | 100 | `guarantee_method`, `auction_stage` |
| 名称 / 标题 | VARCHAR(300) | 300 | `project_name`, `debtor_name` |
| 中等描述文本 | VARCHAR(500) | 500 | `contact_info`, `land_location` |
| 长描述文本 | LONGTEXT | - | `disclosed_defects`, `subject_intro` |
| 超长文本/JSON | LONGTEXT | - | `attachments_json`, `bid_records_json` |
| 时间戳 | DATETIME | - | `updated_at`, `signup_start_time` |
| 日期 | DATE | - | `benchmark_date_norm`, `assessment_date` |
| 金额 | DECIMAL(18,2) | - | `start_price_amount`, `principal_balance_amount` |
| 整数计数 | INT | - | `household_count`, `detail_index` |
| 浮点置信度 | DECIMAL(5,4) | - | `confidence` |
| 展示文本 | VARCHAR(100) | 100 | `start_price_display`, `assessment_display` 类 |

### 7.3 特别说明

- **JSON 字段**：`bid_records_json`、`attachments_json`、`aliases_json` 等使用 LONGTEXT 而非 MySQL JSON 类型，原因是某些来源数据可能不符合严格 JSON 规范。
- **展示文本字段**：以 `_display` 后缀命名字段用于页面展示，保留原始格式（含单位、中文混排），不做类型转换。
- **标准化字段**：以 `_norm` / `_amount` / `_sqm` 后缀命名字段存储标准化数值，用于排序、统计、筛选。
- **金额字段**：统一以"元"为基准进行标准化，`NULL` 表示无法标准化，不写入 `0`。

## 八、去重机制设计

### 8.1 设计目标

同一资产可能多次上拍（一拍流拍后二拍、变卖），每次上拍产生不同 `paimai_id`。未来可能跨平台发现同一资产。需要通过业务字段哈希建立关联。

### 8.2 去重 Hash 计算

在 `asset_dedup_index` 表中维护去重索引，`dedup_hash`（VARCHAR(64)，有索引）由以下各资产类型的核心字段组合计算：

| 资产类型 | 去重字段组合 |
| :--- | :--- |
| debt | `debtor_name` + `creditor` + `principal_balance` + `claim_total` + `asset_location` |
| real_estate | `right_certificate_no` + `asset_location` + `property_location` + `building_area` |
| land | `right_certificate_no` + `asset_location` + `land_location` + `land_area` |
| vehicle | `plate_number` + `vehicle_brand_model` + `storage_location` |
| equipment | `equipment_type` + `storage_location` + `project_name` |
| equity | `target_company` + `equity_ratio` |
| ip | `subject_name` + `right_holder` + `ip_count` |
| goods | `goods_name` + `goods_location` + `project_name` |
| usufruct | `subject_name` + `subject_location` + `right_category` |
| other | `project_name` + `asset_location` |

**计算方法**：
```
dedup_hash = MD5("字段1=值1|字段2=值2|字段3=值3")[:16]
```

字段值先经 `compact_text()` 清洗（去空格、统一全角半角），确保同一资产不同批次的 Hash 一致。

### 8.3 去重查询

通过 `dedup_hash` 查找与当前标的相同资产的其他拍卖记录：

```sql
SELECT d.source_platform, d.source_item_id, d.paimai_id,
       c.project_name, c.auction_stage,
       c.start_price_amount, c.final_price_amount, c.updated_at
FROM asset_dedup_index d
JOIN auction_items_common c ON d.paimai_id = c.paimai_id
WHERE d.dedup_hash = ? AND d.paimai_id != ?
ORDER BY c.updated_at DESC
```

> **注意**：去重不等于阻止入库。即使是同一资产的重复上拍，每条记录也需要独立保存，只是通过 `dedup_hash` 建立关联关系。

## 九、AI 提取流水线设计

### 9.1 提取层次

系统采用三层提取架构，优先级从高到低：

```
第一层：JSON API 结构化提取（detail_json 字段、extendInfoMap）
    ↓ fallback
第二层：HTML 表格/键值对提取（竞买公告 HTML → KVHTMLParser）
    ↓ fallback
第三层：AI 语义提取（百炼 LLM + Vision OCR）
```

每层提取结果记录在 `field_extractions` 表中，标注 `method`：
- `api` — 来自 JSON API
- `rule` / `html_text_regex` — 来自 HTML 规则提取
- `ai` — 来自 AI 文本提取
- `vision_ai` — 来自 AI 图片 OCR
- `derived` — 系统计算（如 `ip_count` = 明细数组长度）
- `not_found` — 所有来源均未找到

### 9.2 证据字段写入规则

每个字段的提取结果写入 `field_extractions` 时，同时写入三段式标准化值：

- `raw_value`：原始提取文本
- `normalized_value`：用于展示和导出的标准化文本
- `numeric_value`：数值/金额类提取 `DECIMAL` 值
- `date_value`：日期类提取 `DATE` 值
- `datetime_value`：时间类提取 `DATETIME` 值
- `value_type`：标记值类型(`text/money/date/datetime`)

### 9.3 知产明细的 AI 图片提取

知识产权标的的逐项明细（软件著作权证书号、专利号等）大多出现在公告附件的图片表格中。系统采用专门的 AI Vision OCR 流水线：

1. **图片获取**：从 `attachments_json` 中筛选 PDF/图片文件
2. **AI Vision 调用**：将图片送至百炼多模态模型，使用专用 Prompt 要求逐行提取
3. **反聚合约束**：Prompt 中明确禁止"计算机软件著作权（7项）""专利权（17项）"等聚合格式，要求每一项必须输出独立 JSON 记录
4. **结果校验**：检测返回结果中是否存在 `(N项)` 聚合模式，若存在则记录 warning 日志
5. **入库**：逐条写入 `asset_ip_details` 表

### 9.4 债权明细的 AI 提取

债权资产包（多户债权）通过 AI 从 HTML 表格或文本中逐户提取，写入 `asset_debt_details` 表，同时自动计算：

- `household_count` = 明细数组长度
- `benchmark_date_norm` = 第一户的基准日（fallback 到整体基准日）

## 十、批次统计与监控

### 10.1 批次状态流转

```
running → success          （全部标的采集成功）
running → partial_success  （部分标的失败，成功率 > 0%）
running → failed           （全部标的失败）
```

### 10.2 监控查询

按批次对比每日采集量与成功率：

```sql
SELECT batch_id,
       started_at,
       status,
       json_extract(summary_json, '$.total_scanned')     AS total_scanned,
       json_extract(summary_json, '$.total_collected')   AS total_collected,
       json_extract(summary_json, '$.success_count')     AS success_count,
       json_extract(summary_json, '$.failed_count')      AS failed_count
FROM crawl_batches
ORDER BY started_at DESC
LIMIT 20
```

### 10.3 跨批次变化检测

通过 `dedup_hash` 对比同一资产在前后批次的变化：

- 新上拍：当前批次中出现，上一批次中不存在
- 重新上拍：之前出现过的资产，以新的 `paimai_id` 再次出现
- 降价变化：同一资产二拍/变卖起拍价低于一拍
- 状态变更：从"竞价中"变为"已成交"或"未成交"

# 项目代码审查待办

本文件用于记录代码审查后需要持续跟进的问题。旧版文件存在编码损坏，已替换为 UTF-8 可读版本。

## 严重问题

1. **敏感信息不得入库或入仓**
   - 模型 API Key、数据库账号密码、浏览器登录态等必须从 `.env` 或本地配置读取。
   - 代码、文档、测试样例中不得硬编码真实密钥。

2. **数据库结构以 MySQL V2 为准**
   - 正式数据写入 MySQL `auction_data`。
   - Runtime storage is MySQL V2 only; SQLite compatibility paths have been removed.
   - Runtime storage is MySQL V2 only; SQLite compatibility paths have been removed.

3. **采集字段必须可追溯**
   - 所有字段结果必须能在 `field_extractions` 或 `raw_payloads` 中找到来源。
   - AI 结果必须携带来源片段，无法提供证据时字段应为空并记录缺失原因。

## 高风险问题

1. **平台 Adapter 需要统一入口**
   - 京东、e 交易、阿里、山东产权、重庆产权、天津产权等平台都应放在 `platform_adapters` 下。
   - `multi_platform_runner` 只负责调度 Adapter，不直接写平台专属解析逻辑。

2. **附件与图片需要结构化存储**
   - 附件、图片、视频统一写入 `item_resources`。
   - 不应只把资源塞进通用字段 JSON。
   - 页面预览应能明确区分附件文件、现场图片、视频和普通页面图片。

3. **AI 补提取应异步执行**
   - 主采集先入库，缺失字段、OCR、附件解析、债权明细和知识产权明细进入队列。
   - 队列任务需要支持重试、失败原因、解锁卡住任务和质量报告。

4. **项目状态和价格口径需要统一**
   - 保留起拍价和最终价。
   - 最终价表示采集时刻可用于展示和统计的价格：未开始取起拍价，进行中取当前价，成交/结束取最终可确认价格。
   - 项目状态优先使用平台接口，其次按开始/结束时间和延时规则计算。

## 中等问题

1. **前端管理页需要保持后台工作台风格**
   - 表格优先，减少装饰性卡片。
   - 所有错误展示后端真实原因。
   - 筛选项包括来源平台、资产类型、项目状态、字段缺失。

2. **数据质量报告需要固定输出**
   - 每次批量采集后生成质量报告。
   - 报告至少包括字段覆盖率、缺失排行、错误样本、平台分布、AI 队列状态。

3. **多平台并发需要可控**
   - 支持多平台并发、单平台多详情并发。
   - 并发数、限速、重试次数必须可配置。
   - 单条失败不应中断整批任务。

## 轻微问题

1. **文档统一 UTF-8**
   - 新增和修改文档必须使用 UTF-8。
   - 避免再次出现乱码片段。

2. **测试覆盖需要持续补齐**
   - Adapter、字段标准化、质量报告、任务调度和 Viewer 页面都需要有最小回归测试。
   - 涉及外部平台的测试应优先使用样本 HTML/API mock。
